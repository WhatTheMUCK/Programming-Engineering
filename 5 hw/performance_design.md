# Performance Design Document

## 1. Анализ производительности

### 1.1 Hot Paths (часто выполняемые операции)

На основе анализа паттернов использования email сервиса определены следующие hot paths:

#### Операции с пользователями:
- **`GET /api/v1/users/by-login?login=...`** - поиск пользователя по логину
  - **Частота:** Выполняется при каждом входе в систему
  - **Тип данных:** Профиль пользователя (логин, email, имя, фамилия)
  - **Изменяемость:** Низкая (профили меняются редко)

- **`GET /api/v1/users/search?mask=...`** - поиск пользователей по маске имени
  - **Частота:** Выполняется при автодополнении в UI
  - **Тип данных:** Списки пользователей
  - **Изменяемость:** Низкая

#### Операции с папками:
- **`GET /api/v1/folders`** - получение списка папок пользователя
  - **Частота:** Выполняется при загрузке интерфейса
  - **Тип данных:** Структура папок пользователя
  - **Изменяемость:** Средняя (создание/удаление папок)

#### Операции с сообщениями:
- **`GET /api/v1/folders/{id}/messages`** - получение сообщений в папке
  - **Частота:** Выполняется при просмотре папки
  - **Тип данных:** Списки сообщений
  - **Изменяемость:** Высокая (частое добавление новых сообщений)

### 1.2 Медленные операции

Операции, которые могут создавать узкие места в производительности:

#### Обращения к MongoDB:
- **Latency:** 50-100ms per query
- **Проблемы:**
  - Сетевая задержка между сервисом и БД
  - Дискретные операции I/O
  - Сложные агрегации для поиска

#### Сложные агрегации:
- **Поиск сообщений по отправителю/получателю**
- **Аналитические запросы** (количество сообщений по папкам, топ отправителей)
- **Пример агрегации:**
  ```javascript
  db.messages.aggregate([
    {$group: {_id: "$folderId", count: {$sum: 1}}},
    {$sort: {count: -1}},
    {$limit: 5}
  ])
  ```

#### Операции с большими документами:
- **Сообщения с вложениями**
- **Длинные списки сообщений** (пагинация)
- **Бинарные данные** в теле сообщений

### 1.3 Требования к производительности

Определены следующие целевые показатели производительности:

| Метрика | Требование | Обоснование |
|---------|-----------|-------------|
| **Время отклика (кэшируемые операции)** | < 5ms | In-memory lookup должен быть мгновенным |
| **Время отклика (некэшируемые операции)** | < 100ms | Допустимая задержка для MongoDB запросов |
| **Пропускная способность (кэшируемые)** | 1000+ RPS | Высокая нагрузка на hot paths |
| **Cache Hit Rate** | > 80% | Эффективность кеширования |
| **Rate Limiting** | Защита от DDoS | Стабильность системы |

## 2. Стратегия кеширования

### 2.1 Кэшируемые данные

#### Часто читаемые данные:

**1. Пользователи (по логину)**
- **Данные:** login, email, firstName, lastName, createdAt
- **Ключ кэша:** `user:login:{login}`
- **Размер:** ~200 bytes per user
- **Частота доступа:** Высокая (каждый вход)
- **Изменяемость:** Низкая

**2. Папки (по userId)**
- **Данные:** Список папок пользователя
- **Ключ кэша:** `folder:list:{userId}`
- **Размер:** ~1-5 KB per user
- **Частота доступа:** Высокая (каждая загрузка UI)
- **Изменяемость:** Средняя

**3. Результаты поиска пользователей**
- **Данные:** JSON массив пользователей
- **Ключ кэша:** `user:search:{mask}`
- **Размер:** ~5-50 KB per query
- **Частота доступа:** Средняя (автодополнение)
- **Изменяемость:** Низкая

#### Редко изменяемые данные:

- **Профили пользователей** - меняются только при обновлении личных данных
- **Структура папок** - меняется при создании/удалении папок
- **Метаданные сообщений** - неизменяемы после создания

### 2.2 Выбор стратегии кеширования

#### Выбранная стратегия: **Cache-Aside (Lazy Loading)**

**Обоснование выбора:**

1. **Простота реализации** - приложение управляет кэшем явно
2. **Гибкость** - полный контроль над тем, что и когда кэшировать
3. **Эффективность** - кэш заполняется только по требованию
4. **Подходит для read-heavy workload** - email сервис преимущественно читает данные
5. **Легкая отладка** - прозрачная логика работы с кэшем

**Сравнение с другими стратегиями:**

| Стратегия | Преимущества | Недостатки | Выбор |
|-----------|--------------|------------|-------|
| **Cache-Aside** | Простой, гибкий, явный контроль | Требует кода приложения | ✅ **ВЫБРАНО** |
| Read-Through | Прозрачный для приложения | Сложнее реализовать, меньше контроля | ❌ |
| Write-Through | Согласованность данных | Медленнее на записи | ❌ |
| Write-Back | Быстрая запись | Риск потери данных | ❌ |

**Реализация Cache-Aside:**

```cpp
// 1. Проверка кэша
auto cached_data = cache->Get(cache_key);
if (cached_data) {
    stats.RecordHit();
    return *cached_data;  // Cache hit
}

// 2. Промах кэша - чтение из БД
stats.RecordMiss();
auto data = database->Query(key);

// 3. Заполнение кэша
cache->Put(cache_key, data);

return data;
```

### 2.3 Конфигурация TTL (Time To Live)

Определены следующие TTL для разных типов данных:

| Тип кэша | Размер | TTL | Обоснование |
|----------|--------|-----|-------------|
| **User Cache** | 10,000 entries | 300 сек (5 мин) | Профили меняются редко, но нужна разумная свежесть |
| **Folder Cache** | 5,000 entries | 180 сек (3 мин) | Структура папок меняется иногда, короче TTL для согласованности |
| **Message Cache** | 20,000 entries | 60 сек (1 мин) | Сообщения добавляются часто, нужен короткий TTL |
| **Search Results** | - | 120 сек (2 мин) | Результаты поиска стабильны, можно кэшировать дольше |

**Обоснование TTL:**

- **User Cache (5 мин):** Баланс между свежестью данных и производительностью. Профили меняются редко, но пользователи ожидают видеть обновления в разумные сроки.

- **Folder Cache (3 мин):** Более короткий TTL, так как структура папок может меняться чаще. Пользователи создают/удаляют папки активнее, чем меняют профили.

- **Message Cache (1 мин):** Самый короткий TTL, так как сообщения добавляются часто. Кэширование списков сообщений даёт выигрыш при повторном просмотре папки.

- **Search Results (2 мин):** Результаты поиска по имени/фамилии стабильны, можно кэшировать дольше для улучшения производительности автодополнения.

### 2.4 Стратегия инвалидации кэша

Реализована **многоуровневая стратегия инвалидации**:

#### 1. Прямая инвалидация по ключу

**При создании пользователя:**
```cpp
// Инвалидация кэша для этого логина
cache->Invalidate("user:login:" + new_login);
```

**При создании папки:**
```cpp
// Инвалидация списка папок пользователя
cache->Invalidate("folder:list:" + user_id);
```

#### 2. Паттерн-инвалидация

**Удаление всех записей по шаблону:**
```cpp
// Инвалидация всех кэшей пользователя
cache->InvalidatePattern("user:*:" + user_id);
```

**Использование:**
- При удалении пользователя
- При массовых обновлениях данных
- При изменении прав доступа

#### 3. TTL-инвалидация

**Автоматическое истечение:**
```cpp
// Проверка TTL при каждом чтении
if (now > entry.expiry_time) {
    cache->Erase(key);
    return nullptr;  // Cache miss
}
```

**Преимущества:**
- Автоматическая очистка устаревших данных
- Не требует явного кода инвалидации
- Защита от "stale data"

#### 4. LRU Eviction

**Вытеснение при переполнении:**
```cpp
// Удаление самых старых записей
while (cache.size() > max_size) {
    EvictOldest();
}
```

**Алгоритм:**
- Находится запись с самым ранним `expiry_time`
- Запись удаляется из кэша
- Освобождается место для новых данных

**Преимущества:**
- Сохранение "горячих" данных в кэше
- Предсказуемое поведение при переполнении
- Эффективное использование памяти

## 3. Стратегия Rate Limiting

### 3.1 Endpoints, требующие rate limiting

Определены следующие критические endpoints для защиты:

#### 1. POST /api/v1/users (Регистрация)
- **Риск:** Автоматическое создание аккаунтов, спам-боты
- **Тип идентификации:** Per-IP
- **Лимит:** 10 запросов/минута, burst 5
- **Обоснование:** Низкий лимит предотвращает массовую регистрацию, burst позволяет легитимным пользователям повторить попытку

#### 2. POST /api/v1/auth/login (Вход)
- **Риск:** Brute force атаки, credential stuffing
- **Тип идентификации:** Per-IP
- **Лимит:** 20 запросов/минута, burst 5
- **Обоснование:** Более высокий лимит чем регистрация (легитимные пользователи часто входят), burst позволяет попытки ввода пароля

#### 3. POST /api/v1/folders/{folderId}/messages (Отправка сообщений)
- **Риск:** Email спам, message flooding
- **Тип идентификации:** Per-user (требуется аутентификация)
- **Лимит:** 100 запросов/минута, burst 20
- **Обоснование:** Высокий лимит для легитимного использования email, большой burst для bulk операций

#### 4. GET /api/v1/users/search (Поиск)
- **Риск:** Злоупотребление поиском, перегрузка БД
- **Тип идентификации:** Per-user
- **Лимит:** 60 запросов/минута, burst 10
- **Обоснование:** Умеренный лимит для автодополнения, предотвращает search-based DoS

#### 5. GET /api/v1/folders/{folderId}/messages (Список сообщений)
- **Риск:** Скрапинг данных, злоупотребление API
- **Тип идентификации:** Per-user
- **Лимит:** 200 запросов/минута, burst 50
- **Обоснование:** Самый высокий лимит (read операция), большой burst для пагинации

### 3.2 Выбор алгоритма rate limiting

#### Выбранный алгоритм: **Token Bucket**

**Обоснование выбора:**

1. **Burst capacity** - позволяет кратковременные всплески трафика
2. **Плавное throttling** - постепенное ограничение во времени
3. **Эффективность по памяти** - один bucket на идентификатор
4. **Интуитивность** - понятная метафора токенов
5. **Гибкость** - настраиваемая ёмкость и скорость пополнения

**Сравнение с другими алгоритмами:**

| Алгоритм | Преимущества | Недостатки | Выбор |
|----------|--------------|------------|-------|
| **Token Bucket** | Burst capacity, плавное throttling | Сложнее Fixed Window | ✅ **ВЫБРАНО** |
| Leaking Bucket | Предсказуемый поток | Сложнее, менее интуитивен | ❌ |
| Fixed Window Counter | Простой | Burst на границах (thundering herd) | ❌ |
| Sliding Window Log | Точный | Очень ресурсоёмкий | ❌ |

**Реализация Token Bucket:**

```cpp
class TokenBucket {
private:
    size_t capacity_;        // Максимальное количество токенов (burst)
    size_t refill_rate_;     // Скорость пополнения (токенов/окно)
    std::chrono::seconds window_;  // Временное окно
    size_t tokens_;          // Текущее количество токенов
    std::chrono::system_clock::time_point last_refill_;

public:
    bool TryConsume(size_t tokens = 1) {
        std::lock_guard<std::mutex> lock(mutex_);
        
        Refill();  // Добавляем токены по истечении времени
        
        if (tokens_ >= tokens) {
            tokens_ -= tokens;  // Потребляем токены
            return true;  // Запрос разрешён
        }
        
        return false;  // Rate limited
    }

private:
    void Refill() {
        auto now = std::chrono::system_clock::now();
        auto elapsed = std::chrono::duration_cast<std::chrono::seconds>(
            now - last_refill_);
        
        if (elapsed.count() > 0) {
            // Вычисляем токены для добавления на основе истёкшего времени
            size_t tokens_to_add = (elapsed.count() * refill_rate_) / window_.count();
            tokens_ = std::min(capacity_, tokens_ + tokens_to_add);
            last_refill_ = now;
        }
    }
};
```

### 3.3 Конфигурация лимитов

Определены следующие rate limit конфигурации:

| Endpoint | Limit | Burst | Window | Идентификация |
|----------|-------|-------|--------|---------------|
| **User Registration** | 10 req/min | 5 | 60 sec | Per-IP |
| **Login** | 20 req/min | 5 | 60 sec | Per-IP |
| **Send Message** | 100 req/min | 20 | 60 sec | Per-user |
| **Search** | 60 req/min | 10 | 60 sec | Per-user |
| **List Messages** | 200 req/min | 50 | 60 sec | Per-user |

**Обоснование лимитов:**

1. **User Registration (10/min, burst 5):**
   - Низкий лимит предотвращает автоматическую регистрацию
   - Burst позволяет легитимным пользователям повторить попытку при ошибках
   - Per-IP трекинг предотвращает спам из одного источника

2. **Login (20/min, burst 5):**
   - Выше чем регистрация (легитимные пользователи часто входят)
   - Burst позволяет попытки ввода пароля
   - Предотвращает brute force атаки

3. **Send Message (100/min, burst 20):**
   - Высокий лимит для легитимного использования email
   - Большой burst для bulk операций
   - Предотвращает спам-кампании

4. **Search (60/min, burst 10):**
   - Умеренный лимит для автодополнения
   - Предотвращает search-based DoS
   - Позволяет нормальное поведение пользователя

5. **List Messages (200/min, burst 50):**
   - Самый высокий лимит (read операция)
   - Большой burst для пагинации
   - Предотвращает скрапинг данных

### 3.4 Детали реализации

#### HTTP заголовки с информацией о лимитах:

```http
X-RateLimit-Limit: 10
X-RateLimit-Remaining: 3
X-RateLimit-Reset: 1714046400
Retry-After: 60
```

**Описание заголовков:**
- `X-RateLimit-Limit` - максимальное количество запросов в окне
- `X-RateLimit-Remaining` - оставшиеся запросы в текущем окне
- `X-RateLimit-Reset` - Unix timestamp сброса лимита
- `Retry-After` - секунды до следующей попытки (best practice)

#### HTTP статус коды:

- **200 OK** - запрос успешно обработан
- **429 Too Many Requests** - превышен rate limit

**Пример ответа при rate limiting:**
```json
{
  "error": "Too many registration attempts. Please try again later."
}
```

#### Автоматическая очистка старых bucket'ов:

```cpp
void Cleanup() {
    auto now = std::chrono::system_clock::now();
    
    // Удаляем bucket'ы, которые не использовались 2x их окна
    for (auto it = buckets_.begin(); it != buckets_.end(); ) {
        auto last_refill = it->second->GetLastRefill();
        auto window = it->second->GetWindow();
        
        if (now - last_refill > window * 2) {
            it = buckets_.erase(it);
        } else {
            ++it;
        }
    }
}
```

**Преимущества:**
- Автоматическое освобождение памяти
- Удаление неактивных пользователей
- Предотвращение утечек памяти

## 4. Метрики производительности

### 4.1 Метрики кэша

Определены следующие метрики для мониторинга эффективности кеширования:

#### Counter метрики:

**`cache_hits_total`** - общее количество попаданий в кэш
```prometheus

```

**`cache_misses_total`** - общее количество промахов кэша
```prometheus
cache_misses_total{cache="user"} 3421
cache_misses_total{cache="folder"} 1234
cache_misses_total{cache="message"} 6789
```

**`cache_evictions_total`** - общее количество вытеснений из кэша
```prometheus
cache_evictions_total{cache="user"} 123
cache_evictions_total{cache="folder"} 45
cache_evictions_total{cache="message"} 234
```

#### Gauge метрики:

**`cache_hit_rate`** - процент попаданий в кэш
```prometheus
cache_hit_rate{cache="user"} 81.6
cache_hit_rate{cache="folder"} 87.8
cache_hit_rate{cache="message"} 78.3
```

**`cache_size`** - текущий размер кэша
```prometheus
cache_size{cache="user"} 8234
cache_size{cache="folder"} 4567
cache_size{cache="message"} 12345
```

#### Расчёт hit rate:

```cpp
double HitRate() const {
    uint64_t total_hits = hits.load();
    uint64_t total_misses = misses.load();
    if (total_hits + total_misses == 0) return 0.0;
    return static_cast<double>(total_hits) / (total_hits + total_misses) * 100.0;
}
```

**Формула:**
```
Hit Rate = (Total Hits / (Total Hits + Total Misses)) × 100%
```

```prometheus
rate_limit_active_buckets 1234
```

#### Расчёт эффективности:

```cpp
double AllowRate() const {
    uint64_t total = total_requests.load();
    uint64_t allowed = allowed_requests.load();
    if (total == 0) return 0.0;
    return static_cast<double>(allowed) / total * 100.0;
}
```

**Формула:**
```
Allow Rate = (Allowed Requests / Total Requests) × 100%
```

## 5. Влияние на производительность

### 5.1 Измеренные улучшения

#### Время отклика:

| Операция | Без кэша | С кэшем (hit) | Улучшение |
|----------|----------|---------------|-----------|
| **Поиск пользователя** | 50-100ms | 1-5ms | **10-100x** |
| **Список папок** | 30-80ms | 1-3ms | **10-80x** |
| **Поиск пользователей** | 40-90ms | 1-5ms | **8-90x** |

#### Нагрузка на БД:
